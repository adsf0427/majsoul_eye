# HUD 元素识别（分数/场风/本场/供托/余牌/动作按钮）— 设计 spec

- 日期：2026-07-04
- 状态：已与用户对齐的设计；Phase A 为本 spec 主体，Phase B 记为后续
- 前置阅读：`docs/DESIGN.md` §2.2/§4（本 spec 是其中"易区 HUD"部分的落地设计，
  但方法从"固定 ROI 裁剪"改为"检测器"，理由见下）

## 1. 背景与关键事实

牌面识别已完成（38 类检测器 mAP50 0.993 + 分类器 99.9%）。剩余场况元素：
四家分数、场风/自风/局数、本场、供托（立直棒）、牌山余牌、动作按钮。

设计被三个**已验证**的事实驱动（样例：`samples/`，手机 2868×1320 与 3168×1440）：

1. **移动端画布比 16:9 宽（AR 2.17–2.20），HUD 重新锚定到物理屏幕边缘**：
   dora 面板/设置/表情/理和鸣切开关条贴边外移，落在"中央 16:9 区域"之外；
   中央信息面板与手牌保持中心锚定；3D 桌面横向延伸视野。
   → "裁中央 16:9 + 固定 ROI"对移动端 HUD 是死路；位置不可赌，**外观可赌**
   （PC/移动 UI 是同一套布局，元素渲染与相对尺寸一致）。
2. **服务器语言不同 → 文字不同**：现有 16 局 captures 是日文 UI（按钮为
   チー/ポン/リーチ…），手机样例为繁体（跳過）与简体（跳过）。字形随语言变，
   语义不变 → 按钮按**语义**定类，各语言字形都是该类的训练样本。
3. **GT 管道现成**：`BoardState` 已逐帧提供 bakaze/kyoku/honba/kyotaku/scores/
   left_tile_count；`GTRecord.raw_liqi` 原样保存 liqi 消息（含 `operationList`）。
   数值/按钮的 WHAT 标签零成本。

明确出范围（用户确认）：分数滚动动画帧不识别；非雀魂官方字体不考虑；
三麻（含拔北按钮）、结算屏、计时器、玩家名 OCR 不在本期。

## 2. 架构总览（两级）

```
任意截图（PC/网页/手机，任意分辨率、任意语言）
   │
   ├─ 全屏 YOLO 检测器 v2（在现有 38 牌类上扩展，一次前向）
   │     38 牌类（含 dora 槽，已有，冻结不动）
   │   + 9 个 HUD 字段类 + 8 个按钮语义类（见 §3）
   │
   ├─ 微读取器（只吃检测到的字段小图，全部转正后为闭集问题）
   │     数字 CTC 读取器（免切分，charset {0-9, -, x, 余}）
   │     round_label 分类器（東/南/西/北 × 1–4局 = 16 类）
   │     风字分类器（4 语义类；東/东 同类）
   │
   └─ 状态组装（纯代码）：类别名→座位/字段，拼装结构化 HUD 状态
```

裁剪路径（两个时刻来源不同）：
- **推理**：ultralytics 把框映射回**原图像素** → 从原分辨率截图直接裁
  （不从 1280 缩略图裁，保小字清晰度）→ 按类别做确定性转正
  （self=0°/across=180°/left/right=±90°，90° 整数倍无损）→ 等比缩到定高 32px。
- **训练**：crop 来自自动标注框本身（不经过检测器）+ **框抖动增广**
  （随机平移/胀缩数个百分点），模拟推理时检测框的不精确。

数字为何不进 YOLO 逐位检测：全屏 1280 输入下中央面板单字约 9×13px，逐位检测
勉强；"检测大字段框 → 微读取器"两级都简单，且第二级标签免费（GT 字符串）。
CTC 免切分：`(crop, "25000")` 即完整训练样本，消灭了投影切分这个唯一脆弱环节
（切分方案留作退路，见 §7 风险）。

## 3. YOLO 新增类别与标注来源（本 spec 核心）

现有 38 牌类**冻结不动**（含 dora 槽牌面——`annotate_frame` 已输出
`dora_boxes`，无需新增）。新增 **17 类**，总计 **55 类**。

每类按本仓一贯公式给出：**WHERE（框怎么来，零手绘）× WHAT（类别/读值 GT 哪来）**。

### 3.1 中央信息面板 7 类（中心锚定，PC/移动渲染一致 → 迁移性最好）

| # | 类名 | 内容 | WHERE（框） | WHAT（读值 GT） | 下游读取器 |
|---|---|---|---|---|---|
| 39 | `score_self` | 自家分数（正立） | 标定 ROI + 逐帧墨迹范围收紧¹ | `scores[hero]` | 数字 CTC |
| 40 | `score_right` | 下家分数（旋转） | 同上 | `scores[(hero+1)%4]` | 数字 CTC（转正后） |
| 41 | `score_across` | 对家分数（180°） | 同上 | `scores[(hero+2)%4]` | 数字 CTC（转正后） |
| 42 | `score_left` | 上家分数（旋转） | 同上 | `scores[(hero+3)%4]` | 数字 CTC（转正后） |
| 43 | `round_label` | 東1局 复合标签 | 标定 ROI（定尺寸，不收紧） | `bakaze`+`kyoku` | 16 类分类器 |
| 44 | `wall_count` | 余64 | 标定 ROI + 墨迹收紧 | `left_tile_count` | 数字 CTC（`余`入 charset，组装时剥掉） |
| 45 | `seat_wind_self` | 面板自家角的风字牌 | 标定 ROI | `(hero_seat−oya)%4` | 4 类风字分类器 |

¹ 墨迹收紧（ink-extent snap）：数字串宽度随数值变（`700` vs `25000` vs 六位），
在种子 ROI 内对发光数字做亮度阈值取外接范围 → 紧框。与 `snap_meld_strip`
同哲学。round_label / seat_wind_self 尺寸恒定，不需要。

分数框旋转方向在标定时实测确定并写死进类别语义（across 必为 180°，
left/right 的 ±90° 方向以实帧为准）。**只检测自家风牌**：自风 + round_label
可推出全部四家风，其余三个角标冗余且更小更斜，不值得三个类。

### 3.2 左上角面板 2 类（PC 上贴左上边缘）

| # | 类名 | 内容 | WHERE | WHAT | 读取器 |
|---|---|---|---|---|---|
| 46 | `riichi_stick_count` | 千点棒 ×N | 标定 ROI + 墨迹收紧 | `kyotaku` | 数字 CTC（`x` 入 charset） |
| 47 | `honba_count` | 百点棒 ×N（本场） | 同上 | `honba` | 数字 CTC |

标定时须实测确认两个图标与 kyotaku/honba 的对应关系（mycv 的
bangzi/benchang ROI 作种子，`coords.py` 已有）。

### 3.3 动作按钮 8 类（语义类，跨语言字形共类）

| # | 类名 | 字形（日/简/繁，都是同一类的训练样本） | WHAT 来源 |
|---|---|---|---|
| 48 | `btn_chi` | チー / 吃 / 吃 | operationList type=2 |
| 49 | `btn_pon` | ポン / 碰 / 碰 | type=3 |
| 50 | `btn_kan` | カン / 杠 / 槓（暗/大明/加杠同一按钮） | type∈{4,5,6} |
| 51 | `btn_riichi` | リーチ / 立直 / 立直 | type=7 |
| 52 | `btn_tsumo` | ツモ / 自摸 / 自摸 | type=8 |
| 53 | `btn_ron` | ロン / 和 / 和 | type=9 |
| 54 | `btn_kyushu` | 九種九牌 / 九种九牌 | type=10 |
| 55 | `btn_skip` | スキップ / 跳过 / 跳過 | 有任一可选操作时随之出现 |

（type=11 拔北为三麻，出范围。type 码按 Akagi/MahjongCopilot 惯例列出，
实现 `state/ops.py` 时以 Akagi 的 liqi 解析器为准逐一核对。）按钮无下游读取器——类别即语义，检测到即
"该动作可用"。Phase A 数据为日文字形；简/繁字形靠 Phase B 采集补充，
类别定义不变。

**按钮的 WHERE（本期唯一没有现成几何的元素）**：
- WHAT 侧新增小提取器 `state/ops.py`：从 `raw_liqi` 解析 `operationList`，
  维护"待决操作区间"——在 seq k 随 ActionDealTile/ActionDiscardTile/
  ActionChiPengGang 提供，被英雄下一动作事件清除；帧落在区间内 ⇒ 按钮可见。
- 框：雀魂按同一优先序把按钮排成一行（锚点/间距确定）。用少量按钮帧标定
  槽位规则（Akagi `majsoul_positions.py` 的 9 槽候选作种子），
  逐帧再做外观校验（局部对比度/轮廓），**检出数 ≠ 待决操作数则整帧丢弃**
  （宁缺毋滥，与 `--drop-violations` 同哲学）。轮廓检测 + 按序赋类作退路。

### 3.4 明确不新增的类

- dora 槽（已有）；立直棒实物/点棒（读计数字段即可，不检测桌面棒子）；
- 其余三家风角标（冗余，§3.1）；名牌/头像/计时器/表情/设置（非场况或出范围）；
- 单个数字**不作为检测类**（两级设计，§2）。

## 4. 微读取器设计

| 读取器 | 输入 | 输出 | 训练数据 |
|---|---|---|---|
| 数字 CTC | 32×W 灰度/彩条（转正后） | 字符序列，charset {0-9, -, x, 余} + blank | 自动标注框裁剪 + 框抖动增广，标签 = GT 数值字符串。~1M 参数 CRNN-CTC 级别 |
| round_label 分类器 | 定尺寸 crop | 16 类（東南西北 × 1–4局） | 同上，标签 = bakaze/kyoku。西/北入极罕见，按类 recall 盯 |
| 风字分类器 | 定尺寸 crop | 4 语义类（東/东 → east） | 同上，标签 = 推导自风 |

组装层（纯代码）：剥 `余`/`x` 前缀 → int；负分保留 `-`（正常局面分数非负，
但 charset 便宜，留着）；按类名归座位；输出结构化 HUD 状态 dict，与
`BoardState` 字段一一对应 → GT 交叉校验即 QA。

## 5. 自动标注管线变更

沿用"一次标定，处处受益"的现有结构，改动集中在：

1. **`coords.py`**：新增/校准 §3 的 HUD 种子 ROI（现有 `REGIONS` 种子来自
   mycv 1080p，逐个在现有 captures 上重标定；标定法沿用 DORA_STRIP/HAND 的
   亮度扫描先例）。
2. **`annotate/frame.py`**：`annotate_frame` 新增 `hud_boxes`（字段框 + 读值
   字符串 + flags），`iter_tile_boxes` 旁新增 `iter_hud_boxes`。
3. **`state/ops.py`（新）**：待决操作提取器（§3.3）。
4. **`build_dataset.py`**：YOLO 标签在 38 类后追加 17 类；同时产出读取器
   训练对 `(crop, string)`（`crops_hud/` 目录，与牌面 crops 并列）。
5. **丢帧谓词**（`is_deal_window` 先例）：
   - 立直宣言后的分数滚动/立直棒动画窗口 → 丢（`last_event=='reach_accepted'`
     附近的帧；实测确定窗口）；用户已确认分数动画帧出范围。
   - 按钮帧检出数不匹配 → 整帧丢（§3.3）。
6. **检测器重训**：55 类 v2，数据 = 现有 16 局全量重建（`--from-annotations`
   路径复用）；kyoku 级切分不变。

## 6. 训练与验收

- **牌面回归门槛**：v2 在既有 held-out（ai_run_8_game1）上 38 牌类 mAP50
  不低于 0.993 − ε（ε=0.005）。不达标 → 退回双模型（HUD 独立小检测器），
  架构其余部分不变。
- **HUD 检测**：新 17 类 mAP50 按类报告；按钮类特别看 recall
  （数据少，见 §7）。
- **读取器**：held-out 帧上"读出字符串 == GT 字符串"精确匹配率，按字段分列
  （目标 ≥99.5%，与牌面 QA 同水位）。
- **端到端**：整帧 HUD 状态与 `BoardState` 完全一致率（一处错即错）。
- 数字类均衡注意：分数串大量是 25000（数字 0/2/5 偏多）；`wall_count`
  0–69 每局扫一遍，天然提供均匀数字分布——读取器训练按字段过采样平衡。

## 7. 数据盘点与风险

| 风险 | 说明 | 缓解 |
|---|---|---|
| **按钮帧存量未知**（最大数据风险） | AI 会话里 MahjongCopilot 秒点按钮，quiet 抓帧可能极少含按钮 | 先跑存量盘点（ops 提取器 × 现有帧）；不足则录 1–2 局人工被动局（决策等待期必然抓到按钮帧）；record_gt 管线现成 |
| 墨迹收紧误框 | 发光/皮肤差异导致阈值失效 | 收紧幅度设上限（不超出种子 ROI）；fill/亮度异常打 flag 丢帧 |
| 槽位规则按钮框不准 | 按钮数不同锚点平移规则与实测不符 | 检出数校验丢帧兜底；轮廓检测退路 |
| CTC 读取器过重/难调 | — | 退路：投影切分 + 逐位 CNN（标签同样免费：切格按序赋 GT 字符，与牌河同招） |
| round_label 西/北入类稀缺 | 正常对局到不了西入 | 标记为已知弱类；Phase B 可用友人场造 |
| 皮肤多样性 | 橙背等皮肤影响墨迹阈值与按钮外观 | Phase A 数据同皮肤，不阻塞；Phase B 随移动端采集顺带扩充 |

## 8. Phase B（后续，另立 spec）

1. **移动端 GT spike（关键未知，先验证再投入）**：Android 模拟器跑雀魂
   手机版 + Akagi MITM（模拟器可设任意分辨率、PC 截屏方便、liqi 同协议）。
   走通 → 移动端标注与 PC 一样免费；走不通 → 手标小 golden set
   （FiftyOne/CVAT 工具链现成）+ 布局随机化增广。
2. **简体/繁体按钮字形**：web 端登 CN 服，同一 capture 管线采集。
3. 训练加强布局增广（mosaic/平移/AR 抖动/多分辨率 letterbox），
   在移动端 golden set 上验收泛化。
4. 平板 4:3 等更窄画布：无样例，待采集后评估。
